# VectorX Functions — Lightweight (pyvx)

<Tier light/> These functions are available in the **Lightweight** execution tier. See [Choosing an Execution Tier](./execution-tiers) for the lightweight vs heavyweight comparison.

The `pyvx` module provides pure-Python MVT (Mapbox Vector Tile) encoding that runs on
**Serverless, standard/shared, and ARM clusters** — no JAR, no init script required.
The two MVT functions it exposes carry the same names as their heavyweight counterparts
(`st_asmvt` and `st_asmvt_pyramid`), so switching tiers is a one-line import change.

:::tip Benchmark & tradeoff
The lightweight (`pyvx`) MVT functions need **no JAR or init script** and are the **only**
option for MVT encoding on Serverless, standard (shared), ARM clusters, and Lakeflow
declarative pipelines. The heavyweight tier (`vectorx`) requires a classic x86 cluster
(JAR). Your compute usually decides the tier. See the [Benchmarking](./benchmarking) page
for timings and methodology.
:::

## Available Functions

| Function | Type | SQL name | Description |
|---|---|---|---|
| `st_asmvt` | Aggregator | `gbx_st_asmvt` | Encode one group of features into a single MVT layer protobuf for one `(z, x, y)` tile. |
| `st_asmvt_pyramid` | Generator (UDTF) | `gbx_st_asmvt_pyramid` | Explode one feature into one row per intersecting `(z, x, y)` tile across a zoom range, MVT bytes already encoded. |

## Options

`st_asmvt` takes a `layer_name` argument (plain string or `Column`). `st_asmvt_pyramid`
additionally accepts:

| Argument | Default | Description |
|---|---|---|
| `layer_name` | `"layer"` | MVT layer name embedded in the protobuf. |
| `extent` | `4096` | MVT tile extent in pixels (MVT v2 standard). |

## Registration

Call `register(spark)` once per session. It uses only `spark.udf.register` and
`spark.udtf.register` — no JVM access — so it works on every runtime including
Serverless and Lakeflow declarative pipelines.

```python
from databricks.labs.gbx.pyvx import functions as vx
vx.register(spark)
```

The one-line swap for the heavyweight tier:

```python
# Lightweight (pyvx) — Serverless-safe, no JAR
from databricks.labs.gbx.pyvx    import functions as vx

# Heavyweight (vectorx) — classic x86 cluster, JAR required
from databricks.labs.gbx.vectorx import functions as vx

# Everything below is identical in both tiers:
vx.register(spark)
```

## st_asmvt

Aggregator that encodes a group of features into a single MVT protobuf blob for one
`(z, x, y)` tile. Each call to `groupBy(z, x, y).agg(vx.st_asmvt(...))` produces the
MVT bytes for exactly one tile.

**Signature:** `st_asmvt(geom_wkb, attrs, layer_name) → BINARY`

**Parameters:**

- `geom_wkb` (`BINARY`) — Feature geometry as WKB in **tile-local coordinates** (pixel
  space, 0..extent). Clip and project each feature to the tile coordinate system upstream
  before calling this aggregator.
- `attrs` (`STRUCT<...>`) — Per-feature attribute struct. Integer, float, boolean, and
  string fields are encoded with native MVT protobuf value types, so numbers arrive at
  downstream clients (MapLibre, Mapbox GL JS, deck.gl) as numbers — not as stringified
  values — enabling data-driven styles and numeric filters.
- `layer_name` (`STRING` or `str`) — MVT layer name. Pass a plain Python string or a
  Column.

**Returns:** `BINARY` — the MVT protobuf for one tile layer. Feed directly into
`gbx_pmtiles_agg` or the [PMTiles Writer](../writers/pmtiles).

**PySpark:**

```python
from databricks.labs.gbx.pyvx import functions as vx
from pyspark.sql import functions as F

vx.register(spark)

# features_df: (z, x, y, geom_wkb BINARY in tile-local coords, name STRING, id LONG)
tiles_df = (
    features_df
    .groupBy("z", "x", "y")
    .agg(
        vx.st_asmvt(
            F.col("geom_wkb"),
            F.struct(F.col("name"), F.col("id")),
            "roads",          # layer name — plain string becomes a literal Column
        ).alias("mvt_bytes")
    )
)
# tiles_df: (z INT, x INT, y INT, mvt_bytes BINARY)
```

**SQL** (after `vx.register(spark)` in the same session):

```sql
SELECT
    z, x, y,
    gbx_st_asmvt(geom_wkb, struct(name, id), 'roads') AS mvt_bytes
FROM features_with_tile_coords
GROUP BY z, x, y
```

## st_asmvt_pyramid

Generator (Python UDTF) that explodes one input feature into one row per intersecting
`(z, x, y)` tile across a zoom range, with MVT bytes already encoded in each row. The
per-tile clip and coordinate transform happen inside the function — no upstream
`ST_Intersection` is needed.

The output schema `(z, x, y, mvt_bytes)` is identical to the heavyweight generator,
so pipelines built against either tier are interchangeable.

**Signature (SQL):** `gbx_st_asmvt_pyramid(geom_wkb, attrs, min_z, max_z, layer_name, extent)`

**Parameters:**

- `geom_wkb` (`BINARY`) — Feature geometry as WKB in **EPSG:4326 lon/lat**. The UDTF
  clips each feature to every intersecting tile and transforms to tile-local coordinates
  internally.
- `attrs` (`STRUCT<...>`) — Per-feature attributes. Same native-typed encoding as
  `st_asmvt`.
- `min_z`, `max_z` (`INT`) — Inclusive zoom range (`0..20`).
- `layer_name` (`STRING`, optional) — MVT layer name; defaults to `"layer"`.
- `extent` (`INT`, optional) — MVT tile extent in pixels; defaults to `4096`.

**Returns:** One row per intersecting tile — schema `(z INT, x INT, y INT, mvt_bytes BINARY)`.

**Caps:** `max_z ≤ 20`; total tile count across the zoom range capped at 10⁶ (matches
the heavyweight generator).

**SQL — LATERAL table function:**

```sql
-- After vx.register(spark):
SELECT t.*
FROM features,
     LATERAL gbx_st_asmvt_pyramid(
         geom_wkb, struct(name, id), 0, 12, 'roads', 4096
     ) t
```

`LATERAL` materializes one `(z, x, y, mvt_bytes)` row per tile the feature intersects.
Each executor processes its partition's features in parallel — tile fan-out is distributed
across the cluster, not serialized on the driver.

**Full pipeline — vector pyramid to PMTiles:**

```python
from databricks.labs.gbx.pyvx    import functions as vx
from databricks.labs.gbx.pmtiles import functions as px
from pyspark.sql import functions as F

vx.register(spark)
px.register(spark)

# Step 1: explode features → per-tile MVT rows (distributed; LATERAL in SQL)
tiles_df = spark.sql("""
    SELECT t.*
    FROM features,
         LATERAL gbx_st_asmvt_pyramid(geom_wkb, struct(name, id), 0, 10, 'roads', 4096) t
""")

# Step 2: aggregate the per-tile MVT bytes → single PMTiles archive
pmt_bytes = (
    tiles_df.agg(
        px.pmtiles_agg(
            F.col("mvt_bytes"), F.col("z"), F.col("x"), F.col("y"),
            '{"name":"roads","attribution":"© My Data"}',
        ).alias("pmt")
    )
    .collect()[0]["pmt"]
)

with open("/tmp/roads.pmtiles", "wb") as fh:
    fh.write(pmt_bytes)
```

For larger pyramids that exceed the Spark cell limit, use the [PMTiles Writer](../writers/pmtiles)
(`pmtiles_gbx` for the lightweight tier) instead of `pmtiles_agg`.

## Native attribute typing

Both tiers encode MVT feature attributes with native protobuf value types:

- **Integer / Long → int64 value**
- **Float / Double → double value**
- **Boolean → bool value**
- **String (and anything else) → string value**

This means downstream clients that consume the MVT tiles receive numbers as numbers and
booleans as booleans — enabling numeric data-driven styles, filter expressions, and
arithmetic in MapLibre GL JS, Mapbox GL JS, and deck.gl without a client-side
`parseFloat` call.

## Composability

`st_asmvt` and `st_asmvt_pyramid` produce `BINARY` MVT bytes that compose naturally with
the PMTiles packaging functions:

- **`gbx_pmtiles_agg`** (UDAF on the [PMTiles Functions](./pmtiles-functions) page) —
  aggregates the per-tile rows into a single `.pmtiles` archive (good for pyramids that
  fit in a Spark cell).
- **[PMTiles Writer](../writers/pmtiles)** (`pmtiles_gbx` lightweight / `pmtiles` heavyweight) —
  streams larger pyramids to a single `.pmtiles` file with no in-memory consolidation.

The raster sibling is [`gbx_rst_xyzpyramid`](./raster-functions#rst_xyzpyramid), which
produces PNG/JPEG/WebP bytes in the same `(z, x, y, bytes)` shape and feeds the same
downstream PMTiles packaging functions.

## Serverless and compute compatibility

| Aspect | Lightweight (pyvx) | Heavyweight (vectorx) |
|---|---|---|
| Install | `%pip install geobrix` (wheel) | Init script + JAR |
| Serverless / shared / ARM | Supported | Not supported |
| Lakeflow declarative pipelines | Supported | Not supported |
| Execution model | Python UDTF / pandas UDF | JVM (Scala + Spark columnar) |
| JVM access | None — `spark.udf.register` only | Required |
