# Benchmarking

GeoBrix ships a benchmark suite that compares the two [execution tiers](./execution-tiers) — **Heavyweight (rasterx)** and **Lightweight (pyrx)** — on the **same inputs**, reporting both **performance** (per-function timing) and **output consistency** (do the two tiers produce the same result?).

This page explains what the benchmark measures, how to run it **on a Databricks cluster** (the primary, same-hardware comparison), how to run it **locally**, how to read the output, and a full table of representative local results.

## What it measures

Each function is timed under two independent models:

- **Pure-core** — the raster operation in isolation: open a tile, call the function, measure. This is the fairest apples-to-apples view of the algorithm itself, with no Spark or serialization in the path.
- **Spark-path** — the registered function (`gbx_rst_*`) applied to a Spark DataFrame of *N* rows. This includes the realistic per-row overhead (UDF dispatch, serialization, Python-worker round-trips for the lightweight tier), and is swept across a **row ladder** (e.g. 10 → 100 → 1,000 → 10,000 rows).

Alongside timing, every pure-core result carries an **output fingerprint** (per-band statistics) so the two tiers can be checked for **consistency**:

- **exact** — every statistic is bitwise-equal across tiers.
- **within_tol** — every statistic agrees to a relative tolerance of `1e-3` **or** an absolute tolerance of `1e-3` (the absolute floor handles near-zero values where a relative comparison is meaningless).
- **divergent** — neither tolerance is met.

The goal of the [one-line tier swap](./execution-tiers#the-one-line-swap) is that results stay consistent; the benchmark is how that guarantee is verified.

:::note
The benchmark is run **from the GeoBrix source repository** (it uses the repo's `gbx:bench:*` commands and a job notebook). A wheel-only install does not include these tools — the cluster benchmark *submits a job to your own provisioned cluster* from a checkout of the repo.
:::

## Running on a cluster

Running on a provisioned cluster is the **true comparison**: both tiers execute on the same hardware, against the same corpus, and the full row ladder and larger tiles are within reach. Results append to a `bench_results` Delta table and a `comparison.csv` / `summary.md` land on the configured Volume.

### Prerequisites

Provision a cluster and stage the artifacts per the [installation guide](../installation):

| Tier | Cluster | Artifacts |
|---|---|---|
| Heavyweight (rasterx) | x86 · DBR 17.3 LTS | Init script + bundle + GeoBrix wheel + the bench tests JAR (`geobrix-*-tests.jar`) |
| Lightweight (pyrx) | x86 **or ARM** | The `[pyrx]` wheel only |

Then fill in the cluster configuration file (`notebooks/tests/databricks_cluster_config.env`) with your cluster ID and Volume paths.

### Run

```bash
# Both tiers, same cluster, full row ladder, all functions
bash scripts/commands/gbx-bench-cluster.sh \
  --cluster-id <your-cluster-id> \
  --run-id cluster-2026-06 \
  --modes both \
  --row-counts 10,100,1000,10000
```

Scale the run with these options:

| Option | Purpose |
|---|---|
| `--modes pure-core \| spark-path \| both` | Which timing model(s) to run (default `both`). |
| `--row-counts 10,100,1000,10000` | The spark-path row ladder — the main scale dimension. |
| `--set core \| full` | Which function set to benchmark — the representative `core` set (default) or every benchmarked function (`full`). |
| `--functions rst_slope,rst_ndvi` | Restrict to specific functions (overrides `--set`). |
| `--warmup` / `--measured` | Warmup and measured iteration counts (defaults 2 / 5). |
| `--lightweight-only` | Skip the heavyweight leg — **required on ARM clusters** (heavyweight is x86-only). |
| `--heavyweight-only` | Skip the lightweight leg. |
| `--no-wait` | Submit the job without blocking on completion. |

```bash
# ARM cluster: lightweight only
bash scripts/commands/gbx-bench-cluster.sh --cluster-id <arm-cluster-id> --lightweight-only
```

Tile sizes, band counts, data types, and projections are set when the corpus is generated (see the local section's scale knobs); the same corpus is reused for both tiers so the comparison is fair.

### Output

- **`bench_results` Delta table** — every measured row (tagged with environment + run ID), so runs accumulate and can be queried/visualized over time.
- **`summary.md` / `comparison.csv`** on the Volume — the human-readable speedup + consistency report (see [Reading the output](#reading-the-output)).

## Running locally

The local pipeline runs the heavyweight tier in the `geobrix-dev` Docker container and the lightweight tier in an isolated Python virtual environment, then compares them. Local runs are intentionally **modest in scale** (small tiles, a single row) — the full row ladder and large tiles belong on a cluster — but they're ideal for a quick, reproducible check.

```bash
# Full local pipeline: generate corpus -> heavyweight -> lightweight -> compare
bash scripts/commands/gbx-bench-all.sh --run-id local-1 --modes pure-core
```

The heavyweight and lightweight legs run **one after the other** (never concurrently) so they don't contend for CPU and skew each other's timings. Outputs land in `test-logs/bench/<run-id>/`.

### Core vs full

`--set` selects how many functions the run benchmarks:

- `--set core` (the default) runs a small, representative set covering each function family — accessors, terrain, band math, warps. It's fast and is the right choice for a routine check.
- `--set full` runs every benchmarked function. It takes longer but gives the complete coverage and parity picture.

An explicit `--functions` list overrides `--set`.

```bash
# Routine check (default): the representative core set
bash scripts/commands/gbx-bench-all.sh --modes pure-core

# Complete coverage: every benchmarked function
bash scripts/commands/gbx-bench-all.sh --set full --modes pure-core
```

Scale and shape the corpus with:

| Option | Purpose |
|---|---|
| `--tile-px 256,512` | Tile sizes (pixels per side). |
| `--bands 1,4` | Band counts. |
| `--dtypes float32,uint16` | Pixel data types. |
| `--srids 4326,32618` | Projections — include a geographic (lat/long) **and** a projected (metre) CRS to exercise both. |
| `--nodata-frac 0.0,0.25` | Fraction of NoData pixels. |
| `--row-counts 10,100` | Spark-path row ladder (keep modest locally). |
| `--modes pure-core \| spark-path \| both` | Timing model(s). |

The individual stages are also available as standalone commands — `gbx:bench:gen-data`, `gbx:bench:heavyweight`, `gbx:bench:lightweight`, and `gbx:bench:compare` — if you want to regenerate just one part of the pipeline.

## Reading the output

The comparison `summary.md` opens with **insights** (biggest wins, the consistency tally) followed by a per-function table:

```text
Consistency (163 compared cells): exact 54 - within-tol 107 - divergent 2

| fn | tile_px | hw_ms | lw_ms | hw_mpix/s | lw_mpix/s | speedup | consistency |
| rst_ndvi | 256 | 654.881 | 0.714 | 0.2 | 187.4 | 917.55 | within_tol |
| rst_slope | 256 | 8.719 | 1.948 | 15.0 | 67.3 | 4.47 | within_tol |
| rst_width | 512 | 0.017 | 0.244 | 31224.3 | 2145.4 | 0.07 | exact |
```

- **`speedup`** is `heavy_ms / light_ms` — greater than 1 means the lightweight tier is faster.
- **`hw_mpix/s` / `lw_mpix/s`** are per-tier throughput (megapixels per second) for each engine; the spark-path tables report `rows/s` instead.
- **`consistency`** is the `exact` / `within_tol` / `divergent` label defined above. Per-cell deltas are in `comparison.csv` (the `max_rel_delta` column).

A per-engine summary is also written for each tier (`heavyweight.summary.md`, `lightweight.summary.md`) with that tier's own timing and throughput in isolation.

### How to read these numbers

- **Band math (`rst_ndvi`/`ndwi`/`nbr`/`evi`/`savi`/`index`/`mapalgebra`)** shows the largest lightweight wins (roughly 240–980×). The heavyweight tier computes these by shelling out to a `gdal_calc` subprocess per call, which dominates its time; the lightweight tier evaluates the expression in-process with NumPy.
- **Reductions, warps, and tiling (`rst_median`/`avg`/`transform`/`to_webmercator`/`retile`/`maketiles`)** favor the lightweight tier by roughly 8–48×.
- **Terrain (`rst_slope`/`aspect`/`hillshade`/`tri`/`tpi`/`roughness`)** is a steadier ~2–5× lightweight win, and all match within tolerance — including on geographic rasters, where both tiers auto-derive the horizontal scale from the CRS.
- **Discrete-grid aggregation (`rst_h3_*`/`rst_quadbin_*`)** is a ~2–6× lightweight win and matches within tolerance.
- **Trivial metadata accessors (`rst_width`/`height`/`numbands` and the world↔raster coordinate helpers)** are sub-millisecond on both tiers; here the JVM-native heavyweight path edges out the Python-worker call, so speedup dips below 1×. At this scale the absolute difference is fractions of a millisecond.
- **Algorithm-bound outliers** — chiefly `rst_viewshed`, and to a lesser degree `rst_contour` and `rst_polygonize` — are slower on the lightweight tier (the heavyweight GDAL implementations are hard to beat), but remain well under a second on these tiles and match within tolerance. A few others (e.g. `rst_gridfrompoints`) sit at or near parity, slightly slower only at the largest tiles.
- **`rst_convolve`** is the single function whose outputs differ: the heavyweight tier applies a GDAL block-halo convolution that no single lightweight boundary mode reproduces exactly. Interior pixels match; only the one-pixel edge ring differs.

### Caveats

- Absolute timings depend on the machine, the corpus, and the tier's install; treat the **relative** speedup and the **consistency** outcome as the durable signal, not the millisecond values.
- Most rows above are **pure-core**, which isolates the algorithm; the `*_agg` aggregators and a few Spark-only operations are timed **spark-path**. The full spark-path model — which you should run on a cluster across the row ladder — adds per-row UDF and serialization overhead and is the better predictor of end-to-end job cost.
- Consistency is checked on output statistics with tolerance, not byte-equality: GDAL and NumPy can differ in the last bits, and neighborhood operations legitimately differ on the one-pixel kernel border. The benchmark treats those as informational, not as divergences.
